## 2. 项目定位

这个项目的核心目标，是沉淀一个用于游戏竞品美术分析的 Codex skill：

`game-art-competitive-analysis`

它面向游戏美术、玩法和商业化包装分析场景，支持把竞品截图或视频帧转化为结构化 Markdown 报告。报告重点覆盖：

- 静态美术风格分析。
- 色彩结构与色板提取。
- UI、图标、按钮、场景、角色等视觉拆解。
- 单竞品深拆与多竞品横向对比。
- 用户调整方向解读。
- 专业调整建议与取舍。
- 差异化和避撞建议。
- AI 美术提示词生成。

从项目内容看，这不是一个普通代码库，而是一个“设计分析工作流工具包”：用 skill 规则、参考框架、输出模板和脚本，把重复性的竞品美术分析过程标准化。

### 阶段一：先做真实案例，验证分析方法

最早的实战样本是 `arrow_kinesic_paths` 报告。它不是抽象模板，而是围绕具体截图做了完整分析：

- 给 6 张图编号。
- 归档图片到稳定目录。
- 按界面类型拆解玩法、通关奖励、额外奖励、大额奖励、双倍奖励、每日奖励。
- 用色板数据支撑色彩判断。
- 识别竞品的商业化包装语言。
- 给出中度调整路线，而不是推翻重做。
- 输出多平台 AI 生图 prompt。

这一阶段的核心价值是验证：“竞品截图 -> 结构化美术结论 -> 可执行调整建议 -> AI prompt”这个链路能跑通。

### 阶段二：把经验抽象成可复用 skill

完成案例后，项目把工作流抽象进 `game-art-competitive-analysis/SKILL.md`。

关键规则包括：

- 默认中文输出。
- 永远输出 Markdown 文档。
- 只分析静态视觉，不分析动画节奏和动效时序。
- 文件型报告必须归档截图，不能依赖临时剪贴板路径。
- 重要结论要绑定截图证据。
- 避免直接克隆竞品，不写要求复刻竞品的 prompt。
- 必须包含差异化和撞脸风险分析。

这些规则明显来自真实工作中的痛点：截图路径容易丢、分析容易主观、prompt 容易写成临摹、报告容易漏掉取舍和后续验证。

### 阶段三：拆分参考资料，减少单文件复杂度

项目没有把所有规则都塞进 `SKILL.md`，而是拆成多个参考文件：

- `analysis-framework.md`：定义静态美术分析维度。
- `style-axis.md`：把“更精品、更年轻、更清晰、更差异化”等用户语言翻译成视觉轴和取舍。
- `prompt-recipes.md`：规定 AI 美术提示词的结构。
- `output-template.md`：规定完整报告、PPT 结构、执行 brief 的 Markdown 输出模板。

这个结构的好处是：

- `SKILL.md` 保持总控。
- 参考文件负责专业深度。
- 后续可以单独扩展色彩、UI、角色、场景或 prompt 模板。
- agent 调用时能按需读取，降低上下文负担。

## 7. 决策与取舍

### 只做静态视觉，不分析动效

取舍：

- 收益：分析边界清晰，截图和视频帧都能处理，报告稳定。
- 代价：无法覆盖动画节奏、转场、特效时序等动态体验。

这个取舍适合当前项目，因为目标是快速生成可复核的美术方向报告。

### 默认输出 Markdown，不直接输出 PPT

取舍：

- 收益：易编辑、易复核、易版本管理，也方便后续再转 PPT。
- 代价：如果直接给外部汇报，还需要额外排版。

对“最小可验证”的设计工作流来说，Markdown 是更稳的中间产物。

### 必须做避撞分析

取舍：

- 收益：降低竞品参考带来的同质化和侵权风险。
- 代价：可能让 prompt 和调整方案更保守，需要更多试图验证。

这符合项目定位：学习竞品的抽象原则，而不是复制其可识别组合。

### 色彩脚本辅助判断，但不替代专业解读

取舍：

- 收益：色彩结论有数据支撑。
- 代价：截图中的黑遮罩、UI 面板、背景大面积颜色会影响综合色板，需要人工解释。

`arrow_kinesic_paths` 报告里已经体现了这一点：综合色板被黑色遮罩影响，所以报告把“遮罩层”和“真实 UI 色彩”分开解读。
